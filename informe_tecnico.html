1. Resumen Ejecutivo del Proyecto

1.1 Descripción del Problema

TechIndustry SpA es una empresa B2B que vende maquinaria y repuestos industriales a través de múltiples sucursales distribuidas a lo largo del país. Cada sucursal genera registros de ventas diarios en archivos CSV que son enviados de forma desincronizada y desordenada a la oficina central. El equipo de Inteligencia Artificial de la organización requiere estos datos para entrenar un modelo de predicción de demanda que permita anticipar el agotamiento de productos críticos, como motores eléctricos y sensores industriales.

El proceso actual presenta tres problemas críticos: (1) un analista consolida manualmente los CSV en Excel, generando un desfase de varios días entre la realización de la venta y su disponibilidad para análisis; (2) no existe trazabilidad de los errores, por lo que un dato incorrecto puede propagarse al modelo sin ser detectado; y (3) se generan silos de información que impiden una visión unificada y en tiempo real del negocio.

1.2 Valor de la Solución: Pipeline Automata-Data

El proyecto Automata-Data construye un pipeline DataOps completamente automatizado, testeable y monitoreable que recorre cuatro etapas secuenciales — Ingesta → Limpieza → Validación → Carga — garantizando que solo datos 100% confiables lleguen al Data Warehouse en la nube (Supabase/PostgreSQL), eliminando el trabajo manual y reduciendo el tiempo de entrega de datos de días a minutos.

La solución entrega un dashboard web operacional construido con Flask que permite ejecutar cada etapa con un clic, visualizar los registros almacenados en tiempo real, consultar los logs de ejecución del día, y realizar operaciones CRUD directamente sobre la base de datos. Esto transforma completamente el flujo de trabajo del equipo analítico, pasando de un proceso frágil y manual a uno automatizado, auditable y escalable.

2. Justificación de la Metodología (PMBOK Híbrido)

Para este proyecto se adoptó un enfoque metodológico híbrido que combina elementos del estándar PMBOK (predictivo) con el marco de trabajo ágil Scrum. Esta decisión se justifica por las características intrínsecas del proyecto: la arquitectura del pipeline y los requisitos de seguridad debían definirse con certeza antes de comenzar a codificar (justificando el enfoque predictivo), mientras que el desarrollo de cada etapa técnica se beneficiaba de ciclos cortos e iterativos con entregables verificables (justificando el enfoque adaptativo).

El pipeline sigue el patrón ELT (Extract → Load con transformaciones previas): cada etapa es un script Python independiente que lee la salida de la etapa anterior, la procesa y escribe su resultado en una carpeta de estado específica, garantizando que ninguna etapa modifique datos de etapas anteriores (inmutabilidad del dato crudo).

4.1 Etapa 1 — Ingesta (ingesta.py)

Propósito: Capturar el archivo CSV de origen y depositarlo en la zona de aterrizaje de datos crudos (landing zone) sin ninguna modificación, preservando la fuente original.

Herramientas: Python 3 stdlib (os, shutil, csv, datetime). La decisión de no usar librerías externas reduce el riesgo de incompatibilidades en distintos entornos de ejecución.

Lógica implementada:

  • Verifica la existencia del archivo fuente (origen/ventas.csv); si no existe, aborta con log de error.
  • Cuenta los registros del CSV mediante un iterador de csv.reader (sin cargar todo en memoria).
  • Crea la carpeta data/raw/ si no existe (os.makedirs(exist_ok=True)).
  • Copia el archivo con un nombre que incluye timestamp en formato ventas_raw_YYYYMMDD_HHMMSS.csv, garantizando idempotencia: múltiples ejecuciones no sobreescriben ingestas anteriores.
  • Registra cada evento en logs/log_YYYYMMDD.txt, acumulando el historial del día.
# Fragmento clave de ingesta.py
fecha = datetime.now().strftime("%Y%m%d_%H%M%S")
destino = "data/raw/ventas_raw_" + fecha + ".csv"
shutil.copy(origen, destino)
escribir_log("archivo copiado a: " + destino)
escribir_log("ingesta terminada ok - " + str(total) + " registros")

Justificación técnica: El naming con timestamp en lugar de sobrescribir el archivo es fundamental para la trazabilidad en DataOps: permite auditar qué datos existían en el sistema en cualquier momento histórico y facilita el re-procesamiento (reprocessing) ante errores en etapas posteriores.

4.2 Etapa 2 — Limpieza y Transformación (limpieza.py)

Propósito: Estandarizar y limpiar el dataset crudo, eliminando inconsistencias que impedirían su análisis correcto, sin modificar el archivo original en data/raw/.

Herramientas: Python 3 stdlib (csv, datetime). Se eligió csv.DictReader sobre pandas para mantener la solución sin dependencias externas y reducir el consumo de memoria en datasets pequeños-medianos.

Transformaciones implementadas:

  • Strip de espacios en blanco: Se aplica .strip() a cada campo antes de cualquier validación, evitando falsos positivos por espacios invisibles.
  • Eliminación de filas con campos vacíos: Si alguna de las 8 columnas obligatorias está vacía, la fila se descarta y se registra en el log con el número de fila.
  • Normalización de texto (Title Case): Los campos producto, categoria, vendedor y region se convierten a Title Case (.title()), garantizando que "pollo entero", "POLLO ENTERO" y "Pollo Entero" se almacenen idénticamente.
  • Normalización de fechas: La función normalizar_fecha() intenta parsear el campo fecha con 4 formatos posibles (YYYY-MM-DD, DD-MM-YYYY, YYYY/MM/DD, DD/MM/YYYY) y los unifica en ISO 8601. Las fechas que no calzan con ningún formato son descartadas.
  • Deduplicación por ID: Se mantiene un set de IDs vistos. Si un ID se repite, la fila se descarta con registro en log.
# Normalización de fecha multi-formato
def normalizar_fecha(fecha_str):
    for fmt in ("%Y-%m-%d", "%d-%m-%Y", "%Y/%m/%d", "%d/%m/%Y"):
        try:
            return datetime.strptime(fecha_str.strip(), fmt).strftime("%Y-%m-%d")
        except ValueError:
            continue
    return None   # retorna None → fila será descartada

4.3 Etapa 3 — Validación Estructural y Semántica (validacion.py)

Propósito: Aplicar un conjunto de reglas de negocio sobre el dataset limpio para garantizar que cada registro es técnicamente correcto y semánticamente coherente antes de ingresar al Data Warehouse.

Herramientas: Python 3 stdlib. La función validar_fila() implementa el patrón validation chain: acumula todos los errores de una fila en una lista y los reporta juntos, permitiendo diagnóstico completo sin detener el proceso ante el primer error.

Validaciones estructurales (tipos de dato):

  • id: debe ser convertible a int.
  • cantidad: debe ser convertible a int.
  • precio_unitario: debe ser convertible a int.
  • fecha: debe tener formato estricto YYYY-MM-DD (post-normalización).

Validaciones semánticas (coherencia de negocio):

  • Fecha no futura: Ninguna venta puede tener fecha posterior a la fecha de procesamiento. Una fecha futura indica un error de captura de datos.
  • Cantidad positiva: cantidad > 0. Una venta de 0 o negativa carece de sentido de negocio.
  • Precio positivo: precio_unitario > 0. Un precio cero o negativo indica error en la fuente.
  • Campos de texto no vacíos: producto, vendedor y region deben tener contenido.

Separación de outputs: Los registros válidos van a data/validated/ y los inválidos a data/invalid/ con una columna adicional errores que describe el problema detectado, facilitando la auditoría y corrección manual.

# Validación semántica: fecha no futura
if fecha and fecha > HOY:
    errores.append("fecha futura: " + row["fecha"])

# Validación semántica: valores positivos
if cantidad is not None and cantidad <= 0:
    errores.append("cantidad debe ser mayor a 0")

4.4 Etapa 4 — Carga al Data Warehouse (carga.py)

Propósito: Transferir los registros validados al Data Warehouse en la nube (Supabase/PostgreSQL) de forma fiable, idempotente y con manejo de errores por lote.

Herramientas: Supabase (PostgreSQL gestionado en la nube), Python 3 urllib.request (cliente HTTP nativo), API REST de Supabase PostgREST. Se eligió Supabase porque ofrece un endpoint REST estándar sin necesidad de instalar librerías de cliente, lo que reduce las dependencias del proyecto.

Mecanismo de carga — Upsert con merge-duplicates: La carga usa el header HTTP Prefer: resolution=merge-duplicates en el endpoint POST de Supabase. Esto implementa un INSERT ... ON CONFLICT DO UPDATE a nivel de base de datos: si el registro ya existe (mismo id), sus campos se actualizan en lugar de generar un error de duplicado. Esto hace la carga idempotente: ejecutar el pipeline dos veces con el mismo CSV produce el mismo estado en la base de datos.

Procesamiento por lotes (batching): Los registros se insertan en lotes de 50 filas. Esto evita timeouts HTTP en datasets grandes y permite reintentar lotes fallidos individualmente sin re-procesar todo el archivo.

# Upsert idempotente por lotes de 50
BATCH = 50
for i in range(0, len(filas), BATCH):
    lote = filas[i:i + BATCH]
    status, error = upsert_filas(lote)   # POST con merge-duplicates
    if status in (200, 201):
        escribir_log("lote " + str(i // BATCH + 1) + " insertado ok")
    else:
        escribir_log("ERROR lote " + str(i // BATCH + 1) + " - " + str(status))

5. Plan de Seguridad DataOps

5.1 Marco Legal Aplicable

El tratamiento de datos en Chile está regulado por la Ley N° 19.628 sobre Protección de la Vida Privada y, de forma especialmente relevante para este proyecto, por la Ley N° 21.719 de Protección de Datos Personales (publicada en 2024, promulgada como modernización integral del régimen chileno alineado con GDPR). Aunque los datos del proyecto son registros de ventas de productos industriales B2B, los campos vendedor y los identificadores de clientes futuros constituyen datos personales bajo ambas leyes. Las obligaciones relevantes incluyen:

  • Principio de finalidad (Art. 3 Ley 21.719): los datos solo se usan para entrenar el modelo de predicción de demanda, no para otros fines.
  • Principio de proporcionalidad y minimización: se capturan únicamente los 8 campos necesarios, sin datos sensibles innecesarios.
  • Principio de seguridad y confidencialidad (Art. 14 Ley 21.719): medidas técnicas y organizacionales para proteger los datos — implementadas en este proyecto vía variables de entorno, TLS, RLS y logging auditable.
  • Principio de responsabilidad proactiva (accountability): el equipo demuestra el cumplimiento mediante el plan documentado en esta sección y la evidencia en logs.

5.2 Políticas Técnicas de Seguridad

PolíticaTécnica AplicadaEstado en el Proyecto
Gestión de Credenciales Las claves de Supabase (URL + API key) se obtienen exclusivamente desde variables de entorno (os.getenv("SUPABASE_URL"), os.getenv("SUPABASE_KEY")) cargadas desde un archivo .env excluido del repositorio mediante .gitignore. Si las variables no están definidas, los scripts carga.py y app.py abortan con un error explícito (fail-closed) en lugar de continuar de forma insegura. El repositorio incluye una plantilla .env.example con los nombres de las variables pero sin valores. ✅ Implementado en carga.py y app.py
Control de Acceso (RBAC) Supabase implementa Row Level Security (RLS) sobre PostgreSQL. Se deben definir políticas RLS que permitan INSERT/UPDATE solo desde el service role, y SELECT solo para usuarios autenticados con rol analyst. ⚠️ RLS activo (detectado en logs: error 42501 en carga anon)
Cifrado en Tránsito Toda comunicación con Supabase ocurre sobre HTTPS/TLS 1.3. La librería urllib.request usa el stack SSL del sistema operativo. El dashboard Flask en producción debe servirse detrás de un proxy HTTPS (Nginx + Let's Encrypt). ✅ HTTPS activo por defecto en Supabase
Cifrado en Reposo Supabase (Fly.io/AWS) cifra los datos en disco con AES-256. Los archivos CSV locales en data/ deben cifrarse con gpg --symmetric o similar si contienen datos sensibles. ✅ Supabase activo / ⚠️ CSV local sin cifrar
Enmascaramiento de Datos Para entornos de desarrollo y testing, los campos vendedor deben reemplazarse por identificadores anónimos (SHA-256(nombre) truncado) para evitar exponer datos personales fuera de producción. ⏳ Pendiente
Auditoría y Trazabilidad Todos los eventos del pipeline se registran en logs con timestamp, incluyendo errores. Los archivos de log deben rotarse semanalmente y almacenarse por al menos 90 días para cumplir requisitos de auditoría. ✅ Implementado en todas las etapas
Validación de Input El dashboard CRUD valida tipos de dato en el formulario HTML (type="number", min, required) y re-valida en el backend Flask antes de llamar a la API de Supabase, previniendo inyección SQL (PostgREST parametriza las queries) y XSS (Jinja2 escapa el output por defecto). ✅ Implementado
Cumplimiento verificado: En el sprint final se ejecutó la migración completa de credenciales a variables de entorno y se incorporó el archivo .env.example + .gitignore al repositorio. La verificación se realiza al arranque de cada script: si SUPABASE_URL o SUPABASE_KEY no están definidas, la ejecución se detiene con un mensaje claro que apunta a la plantilla. Esto cumple con el Art. 14 de la Ley 21.719 (medidas técnicas de seguridad y confidencialidad).

7. Estrategia de KPIs de Monitoreo

Una solución DataOps no está completa sin un sistema de monitoreo que permita detectar degradaciones en la calidad del dato o en el rendimiento del pipeline antes de que impacten los modelos de IA que consumen esos datos. A continuación se definen los KPIs críticos, sus fórmulas de cálculo, umbrales de alerta y los mecanismos de recolección implementados.

KPI 2 — Tasa de Validez
Fórmula: (filas_validas / filas_leidas) × 100
Umbral OK: ≥ 95%
Alerta: < 85%
Fuente: Log de validación
Ejemplo real: 20/20 = 100%
KPI 3 — Tasa de Duplicados
Fórmula: (filas_duplicadas / filas_leidas) × 100
Umbral OK: < 2%
Alerta: ≥ 5%
Fuente: Log de limpieza
Indicador: Problema en la fuente de datos
KPI 4 — Latencia del Pipeline
Fórmula: timestamp(carga_ok) − timestamp(inicio_ingesta)
Umbral OK: < 60 segundos (20 registros)
Alerta: > 5 minutos
Fuente: Timestamps en logs
Ejemplo real: ~16 segundos total
KPI 5 — Registros Cargados a BD
Fórmula: COUNT(*) en Supabase post-carga
Umbral OK: = filas_validas
Alerta: Diferencia > 0
Fuente: Log de carga + API Supabase
Indicador: Fallo en upsert
KPI 6 — Tasa de Error de Carga
Fórmula: (lotes_con_error / total_lotes) × 100
Umbral OK: 0%
Alerta: ≥ 1 lote fallido
Fuente: Log de carga
Indicador: Problema de conectividad o credenciales

8. Conclusiones y Próximos Pasos

8.1 Conclusiones

El proyecto Automata-Data demostró que es técnicamente viable construir un pipeline DataOps funcional con recursos mínimos (Python 3 stdlib + Flask + Supabase), eliminando completamente el proceso manual de consolidación de CSV en Excel. Las cuatro etapas del pipeline — ingesta, limpieza, validación y carga — operan de forma secuencial e independiente, con logging completo en cada paso, lo que proporciona trazabilidad total sobre el ciclo de vida de cada registro de venta.

La ejecución real del pipeline sobre los 20 registros del dataset de TechIndustry SpA resultó en una tasa de completitud del 100% y una tasa de validez del 100%, confirmando que el dataset de prueba fue correctamente preparado. El error inicial de carga (RLS 42501) fue resuelto en el sprint final mediante (a) la migración de credenciales a variables de entorno y (b) la configuración del rol service_role de Supabase para las operaciones de escritura, manteniendo el rol anon únicamente para lectura desde el dashboard.

Desde la perspectiva metodológica, el enfoque híbrido PMBOK/Scrum resultó adecuado: la planificación predictiva garantizó que todos los componentes de la arquitectura se diseñaran consistentemente, mientras que la flexibilidad de Scrum permitió iterar sobre la lógica de limpieza y validación con base en la estructura real de los datos.
